KSAP RPM Platform
Feature Specification

Internal-use Professional Services Management platform. Built via vibe coding. Not for resale. Structured to replace Rocketlane, Smartsheet, and manual Google Sheet workflows. Every feature below maps to KSAP's PRD, evaluation scores, and end-to-end billing lifecycle.

MUST — Blocking. MVP cannot ship without it.
NEEDED — Required within first release cycle.
NICE TO HAVE — Future roadmap. Add when stable.
01 Resource Management Mirrors: Visibility · Forecasting & Planning · Constraints
↳ Visibility
Must

Resource Profile CRUD

Create, read, update, and soft-delete a resource record. Fields: Employee Number, First/Last Name, Department (picklist), Role (picklist), Region (NA/LATAM/APAC), Email, Hourly Cost, Currency of Employment (USD/INR/MXN Peso), Start Date, Engagement Type (Employee/Contractor), Status (Pending/Active/Offboarded). System auto-revokes login and releases future allocations on Offboarded status. Preserves all historical time logs on offboard.

Resource table + modal form
Must

Holistic Bench View

A full-screen grid showing every active resource, their current project assignments, and real-time available hours this week. Color-coded rows: Green = under 80% allocated, Yellow = 80–100%, Red = over 100%. No pagination — designed for a TV-style PMO dashboard wallboard view.

Grid table + color rows
Must

Role-Based Grouping in Bench View

Bench view can be grouped by Department or Role with collapsible group headers showing subtotal available hours per group. Allows PMO to scan headcount by practice area instantly.

Collapsible group rows
Must

Individual Contributor Detail Panel

Clicking any resource opens a side panel showing: current project assignments with hours/%, weekly available hours, pending time-off, and lifecycle status. Hourly cost is hidden from all roles except Finance and Super Admin. Consulting Managers see allocation %, not dollar cost.

Slide-in side panel
Must

Capacity Arithmetic Engine

System continuously calculates each resource's Available_Capacity = 40 hrs/week − Allocated_Hours − Approved_PTO_Hours. This value feeds the Bench View, the Heatmap, and blocks time entry when a date falls outside hire/term dates. A Google Calendar PTO sync reduces capacity to 0 for that day automatically.

Backend calculation engine
↳ Forecasting & Planning
Must

Allocation Heatmap (Resource vs Time)

A scrollable grid: rows = resources, columns = weeks. Each cell shows the allocation % for that week. Cell click reveals which projects contribute. Green/Yellow/Red thresholds enforced. Users can enter allocation as % OR hours — the system bi-directionally converts: Hours = (% ÷ 100) × 8. Over-allocation is allowed with a visible warning badge, never a hard block.

Heatmap grid + tooltips
Must

Future Project Soft Allocation

Allow PMs to allocate resources against Pipeline or Proposed projects (not yet active) to model future capacity impact. These show as a distinct "Forecasted" color in the heatmap vs hard committed allocations. When a project moves to Active, soft allocations auto-convert to hard allocations.

Allocation type toggle
Needed

Availability Search

A search/filter on the bench view: "Show me all resources with Role = OTM Specialist who have ≥ 20 hours available in the week of [date]." Returns a ranked list of matching resources. Used by PMs when staffing new projects.

Filter panel + ranked results
Needed

Skills Matrix / Role Filtering

Each resource profile has a multi-select Skills/Certifications field (e.g., OTM, WMS, QA, Java). The availability search and heatmap can filter by skill tags. Maintained manually by the Super Admin — no complex taxonomy required for MVP.

Tag field + filter chip UI
Needed

Organizational Hierarchy

Each resource has a Reports To field (user lookup). This creates the approval chain: Employee → their Manager → PMO. Used to route time approval to the correct manager for non-project time (Admin, PTO). The hierarchy is visible in a simple org tree view.

Reports-to field + tree view
↳ Constraints
Must

PTO Tracking

Resources or managers can log Time Off entries (Vacation, Sick, Holiday) directly in the platform. A PTO entry for a day sets that day's capacity to 0 and grays it out in the timesheet grid. Contractors cannot log time off. Future: sync with Google Calendar Out-of-Office events automatically.

Time-off modal + calendar block
Must

Holiday Calendars

Admin configures regional holiday calendars (US Federal, India National, Mexico). Holidays automatically block capacity for resources in that region. Holiday days are visually distinct (purple shade) in timesheets and heatmaps.

Admin holiday config table
Must

Over-Allocation Warning (Not a Block)

When a resource is allocated beyond 100% (8 hrs/day), the system saves the allocation with a red warning badge rather than preventing the save. This is intentional — KSAP requires the ability to staff aggressively and manage the conflict manually. A toast notification alerts the PM at save time.

Warning toast + red badge
02 Project Management Mirrors: Portfolio View · Internal Experience
↳ Portfolio View
Must

Project Object CRUD

Full create/edit/delete for a Project. Core fields: Project Name, Customer (lookup), Contract Type (T&M / Fixed Price / Hybrid), Project Type (Staff Aug / OTM Implementation / Migration), SOW Signed Date, SOW Start/End Date, Actual Start/End Date, Project Status (Pipeline / Active / On Hold / Completed / Cancelled), Engagement Model, Payment Terms, Max Hours Cap, Max Amount Cap, Currency, Assigned PM (user lookup), Project Team (team lookup).

Form modal + list table
Must

Portfolio List View

A master table of all projects showing: Project Name, Customer, Status (color pill), PM, Start/End Date, Contract Type, Burn % of budget. Filterable by Status, Customer, PM, Date Range. Default view for Executives shows all; PM default shows only their projects. Each row is clickable to open the Project detail.

Filterable data table
Must

Project Lifecycle State Machine

Projects move through Pipeline → Active → On Hold → Completed / Cancelled. Rules: A project cannot move to Active unless Billing Email and Payment Terms are set on the Customer. A project cannot move to Completed until all time is approved, Actual Completion Date is logged, and a Performance Rating (1–5) is entered. Status changes are audit-logged.

Status dropdown + gate validation
Must

Project Team Assignment

A Project has exactly one Team assigned (Project Team object: Team ID, Project ID, list of Resource IDs). Creating or editing a team is done from within the Project record. Only resources on the team can log time against the project. This is the primary access control for time entry.

Team picker multi-select
Needed

Project Health Status Card

Each project has a health summary showing at minimum: Budget Health (Burn % vs plan), Timeline Health (days ahead/behind), % Complete, Open Risks count, Open Issues count. Health is surfaced as a colored indicator (Green/Yellow/Red) on the portfolio list for executive scan. PM manually sets the narrative status; metrics are auto-calculated.

Status card widget
Needed

Customizable Project Statuses

Super Admin and PMO Manager can add/rename Project Status values (beyond the defaults) to fit evolving KSAP process. For example adding "Hypercare" or "Warranty" stages post-go-live. Status changes always preserve the audit log entry.

Admin status config page
↳ Internal Experience (Work Breakdown Structure)
Needed

Phase Management

A project is broken into Phases (Discovery, Design, Build, Test, Deploy, Hypercare — or custom). Each Phase has: phase_id, name, sequence, start/end date, status (Not Started / In Progress / Blocked / Complete), entry criteria, exit criteria. Phase status auto-rolls up from child Tasks. Phases display as container rows in the WBS grid with a gray background and bold text.

Phase rows in WBS grid
Needed

Task Management

Atomic work unit under a Phase. Fields: task_name, task_type (Configuration/Integration/Data/Testing/Training), owner (user), accountable_role (Internal/Customer), start/due date, estimated_hours, actual_hours, status, dependency_ids, customer_visible flag. Tasks are indented below Phase rows. Users can drag-and-drop a task from one Phase to another. "Log Time" action links directly to the time entry for that task.

Nested draggable task rows
Needed

Milestone Tracking

Contractual checkpoints tied to a project. Fields: name, due_date, approval_required, approved_by, approval_date, status (Pending/Approved/Rejected), Milestone % of Total Cost. For Fixed Price projects, a Milestone reaching Approved status triggers an invoice creation notification to the PM and PMO. Milestones appear on the Gantt as diamond markers.

Diamond markers on Gantt
Needed

WBS Collapsible Tree Grid

The primary project planning UI: a smart spreadsheet where Level 1 = Project header, Level 2 = Phase (gray row, bold), Level 3 = Task (indented, white). Each phase has a chevron to expand/collapse. Budget and hours columns roll up: Task inputs sum into Phase read-only totals. Phase dates auto-derive from earliest task start / latest task end. Planned vs Actual columns side by side.

Collapsible tree grid UI
Needed

Gantt / Timeline View

Split-screen: WBS grid on left, Gantt on right. Phase bars are thick brackets spanning the phase duration. Task bars are thinner, nested under their phase bar. Dependency arrows connect tasks (Finish-to-Start default). Timeline adjusts automatically as actual hours are logged. Zooms: Day / Week / Month. This view is read-heavy — editing happens in the WBS grid.

Split-screen Gantt chart
Needed

RAID Log (Risk, Issue, Change Request)

Three linked objects on every project. Risk: description, probability (1–5), impact (1–5), auto-computed risk_score, mitigation plan, status (Open/Mitigated/Closed). Issue: description, severity (Low/Med/High/Critical), status (Open/In Progress/Resolved). Critical Issue auto-sets Project Health to Red. Change Request: change_type (Scope/Timeline/Cost), impact_analysis, approval workflow. Approved CRs rebaseline the project plan.

Tabbed RAID panel per project
Needed

Financial P&L per Project

A read-only panel on each project showing: Contract Value, Invoiced to Date, Revenue Recognized, Total Labor Cost (resource hours × cost rate in USD via exchange table), Gross Margin % = (Revenue − Cost) / Revenue. Labor costs stored in local currency, converted to USD via Monthly_Exchange_Rate table. Visible to Finance and PM only — hidden from Employee role.

P&L read-only panel
Nice

Project Templates

PMO creates reusable project templates (Phase + Task hierarchy, estimated hours, resource roles, dependencies). When creating a new project, selecting a template pre-populates the full WBS. Resource role placeholders are replaced with actual resource names at assignment time. Templates support up to 5 levels deep for KSAP's typical project structures.

Template library + apply-to-project
Nice

Baseline vs Actuals Tracking

PM can "Baseline" a project at any point in time, taking a snapshot of all planned hours, dates, and budget. The WBS and Gantt then show Baseline vs Current vs ETC in overlay mode. % Complete is auto-calculated from actual hours burned vs planned hours. PM can independently update Estimated-to-Complete (ETC) without altering the baseline.

Baseline snapshot + overlay columns
03 Customer Management No external portal needed — internal record only
↳ Customer Record
Must

Customer Object CRUD

Manage the master customer record. Fields: Customer ID (UUID), Customer Name (unique — duplicate prevention enforced), DBA Name, Status (Active/Inactive/Credit Hold), Billing Address, Tax ID/EIN (encrypted), Payment Terms (Net 15/30/45/Due on Receipt), Currency (USD/MXN/INR), Invoice Delivery Method (Email/Portal/Mail), Billing Email (validated, mandatory before project activation).

Customer form + list table
Must

Contact Management with Functional Roles

Each customer has multiple contacts, each tagged with a Functional Role: Primary Business Contact, Billing/AP Contact (required for QB export), Customer Sponsor/Signatory Authority. At least one Billing/AP Contact must exist before invoicing is enabled. Contacts store: Name, Email, Phone, Role, Status (Active/Inactive).

Contact sub-table with role tags
Must

Financial Governance Rules

Any change to Payment Terms or Billing Address is audit-logged with User ID and timestamp — this is non-negotiable for financial audit compliance. Billing Email and Payment Terms are mandatory before any linked project can move to Active status. The system blocks the project status change and surfaces a clear error message pointing to the missing customer field.

Audit log + gate validation
Must

Customer-to-Project Linkage

Every Project must be linked to exactly one Customer. The Customer record displays a Projects tab showing all associated projects, their statuses, and total invoiced amount. Deactivating a customer (Credit Hold or Inactive) does not delete historical project data — it only prevents new project creation under that customer.

Customer detail with projects tab
Needed

QuickBooks Customer Sync

Bi-directional sync: Customers in QB ↔ Customer records in RPM. When a new Customer is created in RPM, it pushes to QB. When a Customer is updated in QB, it pulls back. The Billing Email and Payment Terms fields are the source of truth in RPM and flow into the QB invoice template. Sync failures surface as an admin alert.

QB API integration layer
04 Time & Reporting Mirrors: Time Tracking · Insights
↳ Time Tracking
Must

Weekly Timesheet Grid (The Core UI)

Primary time entry interface: a grid with rows = Customer + Project combinations, columns = Mon–Sun (weekend columns shaded gray), cells = hours per day. Consultants only see projects they are assigned to via the Project Team object. Users can add rows to select additional projects. Hours must be entered in 0.25-hour increments; any other value is rejected. Daily max = 24 hours hard validation. Standard work week = Mon–Fri, 8 hrs/day, 40 hrs/week.

Timesheet grid — core screen
Must

Description & Billable Flag per Entry

Each time entry cell, when clicked, opens a context panel (side modal) capturing: Description (mandatory text field) and Billable Y/N flag (defaults from project/task config; T&M projects default to Billable = Yes). Submitting or saving without a description is blocked. If the user overrides the default Billable flag, a visual indicator appears for the Approver's attention.

Context panel slide-in
Must

Time Entry Validation Rules

Hard-blocked scenarios: (1) Time logged against a project with status Closed, On Hold, or Archived. (2) Entries on dates before the user's Hire_Date. (3) Entries on dates after the user's Termination_Date. (4) Missing Description on any entry. (5) Hours not in 0.25 increments. (6) More than 24 hours in a single day. All blocks surface a clear inline error message — never a silent failure.

Inline validation errors
Must

Smart Pre-population

When a consultant opens a new week's timesheet, the system auto-populates rows with the same Customer + Project combinations from the previous week, with 0.00 hours. This eliminates repetitive row selection for ongoing projects. The consultant can delete pre-populated rows they won't use this week. New project rows can still be added manually.

Auto-pre-populate from prior week
Must

Approval Workflow State Machine

Timesheet states: Draft → Submitted → Manager Approved → Finalized (Finance) → Locked/Invoiced. Rejected returns to Draft. Rejection requires a mandatory Rejection Reason text from the manager — displayed in Red with a tooltip on the consultant's grid. Contractors cannot log PTO. Locked timesheets require Super Admin override to edit, which writes to the System Audit Log.

State machine + approval queue
Must

PMO-Sanctioned Time Entry Categories

A PMO Manager–maintained list of approved time categories (e.g., "Development," "Project Management," "Travel," "Training"). Each category has a default Billable/Non-Billable flag. Consultants select from this list when logging time. The list is managed via an admin config screen, not hardcoded. Consultants can override the default flag on individual entries.

Admin category config + dropdown
Must

Timesheet Compliance Workflow

Automated non-compliance enforcement: Friday reminder notification to all consultants who have not submitted their timesheet. If not submitted by Monday noon, a notification goes to the consultant, their PM, and PMO. A "Non-Compliance" flag appears on the resource's profile for that week. This replaces the current manual follow-up process entirely.

Scheduled notification engine
↳ Insights & Reporting
Must

Time Explorer Export (CSV)

A dedicated export tool for Finance and PMs. Filters: Date Range, Group By (Employee or Project), Status Filter (Draft/Approved/Locked/All). CSV output columns: Entry_ID, Date (YYYY-MM-DD), Project_Name, Task_Name, Resource_Name, Hours (0.00 format), Billable_Flag, Description, Status, Approver_Name. This directly replaces the current Google Sheet time import workflow.

Export modal + CSV download
Must

Project Burn Report

Per-project view showing: Total Budget vs Actual Spend (hours × bill rate for T&M; milestone value for Fixed Price), weekly burn rate, projected completion date based on current velocity. Triggered burn rate alert: if Actual_Burn exceeds Planned_Burn by >15% for two consecutive weeks, a red alert badge appears and the PM receives a notification.

Burn chart + alert badge
Must

Utilization Reports

Two key metrics for the executive dashboard: Total Utilization = Total Logged Hours / Total Available Hours. Billable Utilization = Billable Hours / Total Available Hours. Filterable by date range, department, region, and resource. Displayed as both a summary stat card and a trend line chart over time. Data source = locked/approved timesheet records only for accuracy.

Stat cards + trend chart
Needed

Planned vs Actual vs ETC Dashboard

A cross-project view comparing Planned Hours (from project WBS), Actual Hours (from approved timesheets), and Estimated-to-Complete (PM-maintained field) per project. Shown as a bar chart with three grouped bars per project. Filter by PM, Status, Date. This answers the executive question: "Are we on track across the portfolio?"

Grouped bar chart + table
Needed

Project Margin Report

Finance-only view: Gross Margin % = (Revenue − Labor Cost) / Revenue per project, aggregated across the portfolio. Labor cost uses local currency converted to USD via Monthly_Exchange_Rate table. Shows historical margin trend by project. Exportable to CSV. This is the primary financial health signal for leadership.

Margin table + export
Nice

Role-Based Default Dashboard Views

When a user logs in, they land on a dashboard pre-configured for their role: Executive sees portfolio health + margin. PM sees their projects + team allocation. Consultant sees their timesheet + tasks. Finance sees pending billing queue + invoices. Users can customize and save multiple additional dashboard views without losing their role default.

Configurable widget dashboard
Nice

Scheduled Report Distribution

PMO can configure a report (e.g., Weekly Project Status, Timesheet Compliance) to be automatically emailed on a schedule (every Friday at 5pm) to a defined list of users or user groups. Reports are delivered as PDF attachments or inline HTML emails. This replaces Brian's manual weekly oversight spreadsheet distribution.

Scheduled email engine
05 Invoicing & Financial Management End-to-end billing lifecycle — QB is the AR system of record
↳ Invoice Lifecycle
Must

T&M Invoice Generation from Approved Time

PM can generate a Draft Invoice from all approved, un-invoiced time entries for a project within a billing period. Invoice line items display: Task Name, Resource Name, Date, Hours, Rate, Amount, Description (PM can edit/sanitize descriptions before sending). Invoice can be regenerated (deleted and recreated) until it is formally Approved. Once Approved, it locks.

Invoice builder from time entries
Must

Fixed Price / Milestone Invoice Creation

For Fixed Price projects, invoices are manually created by PM when a Milestone is Approved. The system auto-populates: Customer, Billing Address, Payment Terms, Milestone Name, Milestone Amount (% of contract value). PM reviews, approves, and pushes to QB. Completion of a Milestone triggers a notification to PM and PMO that an invoice is ready to be cut.

Manual invoice + milestone trigger
Must

Invoice Approval Workflow

Invoice states: Draft → PM Approved → Finance Approved → Sent to QB → Paid. PM approves their own invoices. PMO/Finance can approve any invoice. Once Finance Approved, invoice pushes one-way to QuickBooks for AR. Payment status flows back from QB (Paid/Partially Paid/Overdue) into the platform's invoice record. Locked invoices require Super Admin to unlock.

Invoice state machine + QB sync
Must

QuickBooks Invoice Push (One-way)

Upon Finance Approval, the invoice is automatically pushed to QuickBooks Online as an Invoice record, mapped to the correct QB Customer and GL accounts. No manual re-entry. Finance does not need to touch QB for standard invoices — the platform handles the push. Failed pushes surface as an admin alert with the QB error message for manual resolution.

QB API one-way push
Needed

Payment Tracking on Invoice

Once an invoice is sent, the platform tracks payment records against it: Payment Date, Amount Paid, Check/Reference Number. Supports partial payments (multiple payment entries per invoice). Outstanding balance is calculated automatically. Invoice status updates: Paid / Partially Paid / Overdue. Payment data syncs back from QB or can be entered manually.

Payment sub-records on invoice
Needed

Pending Billing Queue

A Finance-specific view listing all projects with approved, un-invoiced time that is ready for billing. Columns: Customer, Project, Billing Period, Unbilled Hours, Estimated Invoice Amount, Days Since Last Invoice. Finance uses this to prioritize monthly billing runs. Clicking a row initiates invoice generation for that project.

Finance billing queue view
06 Security & Access Control Mirrors: Customer Collaboration → Security · Role-based Access Control
↳ Role-Based Access Control (RBAC)
Must

Role Definitions & Permission Matrix

Six roles enforced across the system: Super Admin (system-wide config, user management, unlock records), Executive (read-all, dashboards, profitability), Consulting Manager (manage resources, approve PTO — cannot see salary/cost unless authorized), Project Manager (write own projects, approve time, manage budget), Employee/Consultant (log own time, view own assignments), Finance (invoicing, cost rates, QB sync, reconciliation).

Role config + permission matrix
Must

Data-Level Security by Role

Critical data isolation: Hourly Cost Rate — visible to Finance and Super Admin only. Rate Cards and billing rates — visible to PM and above only. Project financials (margin, P&L) — visible to Executive, Finance, Super Admin. Other consultants' time entries — PM sees only their assigned projects; Employee sees only their own. All restrictions enforced at the API level, not just the UI.

Row/field level data isolation
Must

System Audit Log

Every sensitive action is written to an immutable audit log: Payment Terms changes, Billing Address changes, record unlocks, Admin overrides, invoice status changes, user role changes. Each log entry records: User ID, Timestamp, Action Type, Old Value, New Value, Entity ID. Finance and Super Admin can search and export the audit log. Log entries cannot be deleted.

Audit log table + search
Must

Multi-Tenancy (Customer Data Isolation)

If external customer users are ever granted access (future), they can only see data explicitly shared with them. For now (internal use), this means project data scoped by Project Team membership: a consultant sees only projects they are assigned to. A PM sees only their own projects unless granted cross-project access by Super Admin. No consultant can ever browse another consultant's time entries.

Scoped data access by team
07 Improvement Areas & Future Roadmap Mirrors: Manual Process Reduction · System Consolidation · AI Integration
↳ Manual Process Reduction (Phase 2)
Nice

CRM / Opportunity Management

Track opportunities from inception through closing. Fields: Stage (Lead → Proposal → Won → Lost), win probability %, estimated revenue, expected start date. Won opportunity converts directly into a Customer + Project record without re-entry. Contact → Opportunity → Customer conversion lifecycle. A won opportunity can seed an initial project plan from the sales intake details. Replaces Zoho for pipeline visibility.

CRM pipeline + conversion flow
Nice

Google Calendar PTO Sync

System ingests Google Calendar "Out of Office" events for each resource. An 8-hour OOO event on a day reduces Available_Capacity to 0 for that day, automatically updating the heatmap. Prevents a PM from scheduling a resource against a project on a day they're OOO. PTO events created in the platform optionally push back to Google Calendar.

Google Calendar API integration
Nice

Document Attachments

Attach SOWs, Change Requests, design documents, and emails directly to a Project or Customer record. Stored links to Google Drive or SharePoint (not file storage within the platform). Attachment entry records: File name, Link URL, Uploaded By, Upload Timestamp, Document Type. Viewing requires the linked Google/SharePoint permission — the platform only stores the reference.

Attachment records + Drive links
Nice

Multi-Currency Exchange Rate Table

Admin-maintained Monthly_Exchange_Rate table: Currency pair (INR→USD, MXN→USD), Month, Rate. System uses this table to convert all resource cost rates to USD for unified margin reporting. Exchange rates are entered manually by Finance once per month. All financial reports display in USD regardless of the resource's local currency of employment.

Exchange rate config table
↳ AI Integration (Phase 3 — Vibe-coded add-ons)
Nice

AI Burn Rate Alert Engine

Daily evaluation: if Actual_Burn exceeds Planned_Burn by >15% for two consecutive weeks, the system automatically flags the project as "At Risk - Budget" and sends a structured alert to the PM and Executive: "Project X burn rate is 22% over plan for 2 consecutive weeks. Predicted budget exhaustion: [date]." No manual monitoring required.

Automated alert + prediction text
Nice

AI Stagnation Alert

If a project with Active status has zero hours logged for 14 consecutive days, the system triggers a stagnation alert to the PM and PMO: "Project Y has had no time logged for 14 days. Action required." This catches forgotten or abandoned projects before they become a billing or compliance issue.

14-day inactivity detection
Nice

AI Executive Status Summary

A weekly auto-generated plain-English summary per project: "Project A — 73% complete, on budget, 2 open risks (medium). On track for go-live March 31." Aggregated into a portfolio-level brief for the Executive role. Generated by calling the Anthropic API with structured project data as context. PM reviews and can override/supplement the AI narrative before it is distributed.

LLM-generated status brief
