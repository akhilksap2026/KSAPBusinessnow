## Bucket 4 — Project & Task Module Enhancements
**Sprint 4 · Estimated: 3–4 days**

> Depends on Bucket 1 (schema — parentTaskId, etcHours, task_assignments,
> task_comments, saved_filters) and Bucket 2 (labels).
> Do NOT touch /resources, /timesheets, /customers, or /opportunities pages.

```
You are enhancing ONLY the Projects module (/projects, /projects/:id)
and the standalone Tasks page (/tasks).
Do NOT change any other pages or API routes.
Do NOT break existing project list, project detail tabs, or milestone functionality.
All changes are additive.

─── 1. API: TASK HIERARCHY ENDPOINTS ───
In api-server/src/routes/tasks.ts:

GET /api/tasks — update to include parentTaskId, hierarchyLevel, isLeaf, etcHours,
  commentCount in the returned fields.
  Add support for ?parentTaskId= query param to fetch children of a specific task.

GET /api/projects/:id/full — update to return tasks in hierarchical order:
  {
    ...project,
    taskTree: [
      {
        ...task,
        children: [
          { ...task, children: [...] }
        ]
      }
    ]
  }
  Build this tree by fetching all tasks for the project, then nesting them
  by parentTaskId. Tasks with parentTaskId=null are roots.

POST /api/tasks — update to accept parentTaskId, etcHours.
  When a task has children (i.e., becomes a parent), set isLeaf=false.
  New leaf tasks default to isLeaf=true.
  Auto-calculate hierarchyLevel from parent chain.

PATCH /api/tasks/:id — update to accept etcHours.
  IMPORTANT: etcHours is stored directly — never derive it as estimatedHours - actualHours.

─── 2. API: TASK COMMENTS ENDPOINTS ───
Create api-server/src/routes/task-comments.ts:

GET /api/task-comments?taskId=:id
  Returns all comments for a task, ordered by createdAt ASC.
  Join with resources to include authorName, authorTitle.

POST /api/task-comments
  Body: { taskId, authorId, body, isExternal? }
  Parse @mentions from body: any @word that matches a resource name
  → populate mentionedUserIds with their resource IDs.
  After inserting comment: increment tasks.commentCount by 1.
  For each mentionedUserId: create a notification record:
    { userId: mentionedUserId, type: 'mention', title: '[Author] mentioned you on [task name]',
      body: first 100 chars of comment body, resourceId: taskId, resourceType: 'task' }

PATCH /api/task-comments/:id — update body, updatedAt
DELETE /api/task-comments/:id — decrement tasks.commentCount

Register these routes in api-server/src/routes/index.ts.

─── 3. API: SAVED FILTERS ENDPOINTS ───
Create api-server/src/routes/saved-filters.ts:

GET /api/saved-filters?context=tasks
  Returns: personal filters (ownerId = current user) + shared filters (isShared=true)
  ordered by name.

POST /api/saved-filters
  Body: { name, ownerId, isShared, filterContext, filterJson }

DELETE /api/saved-filters/:id
  Only allow if ownerId matches current user.

Register in routes/index.ts.

─── 4. API: TASK ASSIGNMENTS ENDPOINTS ───
Add to api-server/src/routes/tasks.ts:

GET /api/task-assignments?taskId=:id
  Returns all assignments for a task with resource name, title, roleOnTask, estimatedHours.

POST /api/task-assignments
  Body: { taskId, resourceId, roleOnTask?, estimatedHours? }

DELETE /api/task-assignments/:id

─── 5. API: TASK DEPENDENCIES ───
Update api-server/src/routes/tasks.ts (or task-dependencies.ts):
GET /api/task-dependencies?taskId=:id
  Returns both: predecessors (where successorTaskId = :id) AND
                successors (where predecessorTaskId = :id)
  Include the lagDays field (from Bucket 1 schema change).

POST /api/task-dependencies
  Body: { predecessorTaskId, successorTaskId, dependencyType, lagDays? }
  dependencyType must be one of: 'FS'|'SS'|'FF'|'SF'

DELETE /api/task-dependencies/:id

c) Actions column: add "Copy" and "Save as Template" icon buttons.
   Copy: POST /api/projects/:id/copy (see endpoint below)
   Save as Template: POST /api/projects/:id/save-as-template (see endpoint below)

─── 7. API: COPY PROJECT + SAVE AS TEMPLATE ───
In api-server/src/routes/projects.ts:

POST /api/projects/:id/copy
  Duplicates the project record (with name "[Original Name] — Copy")
  Also duplicates all milestones for the project.
  Also duplicates all tasks (preserving parentTaskId relationships).
  Returns the new project's id.
  On success, frontend navigates to /projects/:newId.

POST /api/projects/:id/save-as-template
  Body: { templateName, description }
  Reads the project's milestones and tasks.
  Creates a project_templates record with templateData containing
  the full task and milestone structure (as JSON, with IDs replaced by
  sequential slugs so they're reusable).
  Increments usageCount on the template.
  Returns the new template's id.

─── 8. PROJECT DETAIL: TASKS TAB — HIERARCHICAL TREE ───
Replace the existing flat task list in the Tasks tab with a collapsible tree:

a) Tree Structure:
   - Root nodes (hierarchyLevel=0) = Phase rows, styled with a slightly darker
     background (e.g., bg-slate-100).
   - Child nodes are indented 20px per level.
   - Each row has a ChevronRight / ChevronDown toggle if it has children.
   - Collapsed rows only show the parent row.

b) Parent Row Data (rolled up from children):
   - Total estimatedHours: sum of all leaf descendants
   - Total actualHours: sum of all leaf descendants
   - Completion %: weighted average of leaf completion %
   - Assignee count: count of unique assignees across children
   - These are computed client-side from the taskTree data.

c) Leaf Row Data (isLeaf=true):
   Shows: task name | assignee avatar stack (up to 3) | status badge |
          priority badge | estimatedHours | actualHours | ETC | due date |
          comment count badge | progress %

     "Time can only be logged on subtasks")
