─── 3. ROLE AUTO-POPULATION ON TIMESHEET POST ───
In api-server/src/routes/timesheets.ts, update POST /api/timesheets:
  After receiving the body, before inserting to DB:
  1. Look up the resource's allocation for this project:
       SELECT allocationType FROM allocations
       WHERE resourceId=:resourceId AND projectId=:projectId
       LIMIT 1
  2. Look up the project's rate card:
       SELECT rc.id FROM rate_cards rc
       WHERE rc.projectId=:projectId
       OR (rc.projectId IS NULL AND rc.isTemplate=true)
       ORDER BY rc.projectId DESC NULLS LAST
       LIMIT 1
  3. Look up the resource's defaultRole and find a matching rate_card_role:
       SELECT sellRate, buyRate FROM rate_card_roles
       WHERE rateCardId=:rateCardId AND roleName=:defaultRole
  4. If found: set billedRole=defaultRole, sellRate=sellRate, costRate=resource.costRate
  5. If not found (role not on rate card): set billedRole=defaultRole,
     sellRate=null, costRate=resource.costRate
  This auto-population is silent — the user never sees these fields in the UI,
  but they are stored for margin calculations.

  (All / Submitted / Approved / Rejected)

    Checkbox | Resource | Date | Project | Task | Hours | Billable |
    Daily Comment | Billed Role | Status badge | Actions

  Status badges:
    submitted=blue, approved=emerald, rejected=red

  Bulk actions bar (appears when checkboxes selected):
    "Approve Selected" → POST /api/timesheets/approve { timesheetIds, approverId }
    "Reject Selected" → opens a dialog for rejection reason →
      POST /api/timesheets/reject { timesheetIds, approverId, reason }

  API endpoints needed in api-server/src/routes/timesheets.ts:
    POST /api/timesheets/approve — accepts { timesheetIds: number[], approverId: number }
    POST /api/timesheets/reject  — accepts { timesheetIds: number[], approverId: number, reason: string }
